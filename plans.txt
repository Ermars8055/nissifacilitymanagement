Comprehensive System Architecture: Zero-Budget Anti-Spoofing & Location Verification Platform

1. Core Engineering Philosophy
The platform is designed to be hardware-agnostic and zero-infrastructure cost.
Instead of relying on absolute physical measurements (like uncalibrated phone barometers) or expensive infrastructure (like Bluetooth beacons or per-floor hardware gateways), the system enforces security through a combination of Geospatial Math, Network Context, and Temporal Speed Traps (Time-of-Flight Tracking).
The core objective is to make fraud humanly and mathematically impossible without requiring the client to spend a single dollar on extra equipment.



2. Component 1: On-Site Attendance & Arrival (Horizontal Verification)
To prevent workers from clocking into their shift from home or while commuting, the platform uses standard GPS horizontal tracking paired with strict anti-spoofing software flags.

The Mechanism
1. The Target Anchor: The building manager registers the exact center point of the building during onboarding. The database saves these static values as target_lat and target_lng.
2. The Haversine Mathematical Check: When a worker scans the attendance QR code, the mobile app captures the device's coordinates (user_lat, user_lng). Because concrete roofs cause GPS signals to drift, the server cannot look for an exact string match. It must calculate the true spherical distance in meters using the Haversine Formula:
\(d=2R\cdot \arcsin \left(\sqrt{\sin ^{2}\left(\frac{\Delta \phi }{2}\right)+\cos (\phi _{1})\cos (\phi _{2})\sin ^{2}\left(\frac{\Delta \lambda }{2}\right)}\right)\)
Where:
* \(R\) is the Earth's radius (\(6,371,000\text{ meters}\)).
* \(\phi_1, \phi_2\) are the latitudes in radians.
* \(\Delta \phi, \Delta \lambda\) are the differences in coordinates in radians.

The Security Flags
* Geofence Threshold: The server compares the calculated distance against a strict threshold (e.g., \(40\text{ meters}\)). If the user is further away, the request is rejected.
* Mock Location Intercept: To stop users from downloading "Fake GPS" applications from the Play Store, the mobile app frontend queries the operating system API for the isMockProvider flag. If a location-spoofing app is active, the app blocks the payload before it ever reaches the backend.



3. Component 2: Floor & Asset Tracking (Vertical Verification via Time-of-Flight)
Because Floor 1, Floor 2, and Floor 3 sit directly on top of each other, they share the exact same GPS coordinates. To prove a worker physically climbed to a higher floor without utilizing uncalibrated hardware barometers, the system implements Time-of-Flight (ToF) Velocity tracking.

The Concept
Time-of-Flight serves as a software-driven speed trap. It measures the physical transit seconds elapsed between two specific scanning nodes to ensure a worker did not teleport.

[Lobby QR Scanned] ───► (User travels via Lift or Stairs) ───► [Floor 3 Asset Scanned]
   Timestamp: T1                                                   Timestamp: T2
   
   Verification: If (T2 - T1) is too fast for human locomotion, REJECT.

The Real-World Locomotion Windows
Human vertical transit speeds are bound by the laws of physics. The server maps out a "Humanly Possible Window of Time" for each high-floor asset relative to the Ground Floor Lobby:
* The Lift Boundary (Fastest): A standard office elevator travels at roughly \(5\text{ seconds per floor}\), requiring an additional \(10\text{ seconds}\) for door cycles. Reaching Floor 3 safely takes a minimum of \(25\text{ to }30\text{ seconds}\).
* The Stair Boundary (Average): An average adult climbs standard building stairs at roughly \(20\text{ seconds per floor}\), totaling \(60\text{ to }90\text{ seconds}\) for Floor 3.
* The Inactivity Boundary (Maximum): A 30-minute hard expiration token is applied. If a worker takes longer than 30 minutes, the session is invalidated, forcing them to re-establish their baseline at the lobby.

Catching the "Pocket Scan" Loophole
If a rogue employee takes a photo of the Floor 3 QR code, places it in their pocket, stands on the ground floor, and scans the Lobby code followed immediately by their pocket print, the time delta (\(T_2 - T_1\)) will read between \(1\text{ and }5\text{ seconds}\). Because no lift on earth can cross 3 floors of concrete in under 25 seconds, the server flags the transaction as fraudulent and denies verification.



4. Component 3: Continuous Asset Chains & Breadcrumb Logic
Workers managing physical building assets often have dozens of checkpoints or computers to audit on a single shift. Forcing them to go all the way back down to the ground floor lobby to reset their timer for every single asset would cause immense user friction and destroy operational efficiency.
To solve this, the software uses a Dynamic Breadcrumb Chain.

[Lobby Entry] ──► [Floor 1: PC-1] ──► [Floor 1: PC-2] ──► [Floor 2: PC-3] ──► [Floor 3: PC-5]
  (Reset Token)      (ToF Check 1)      (ToF Check 2)      (ToF Check 3)      (ToF Check 4)

The Chain Link Architecture
1. The Entry Check-in: The worker scans the master Ground 0 Lobby QR once. This initializes their active field session.
2. The Rolling Baseline: When the worker walks to Floor 1 and scans PC-1, the server checks the time against the Lobby scan. If approved, the server updates the rolling baseline timestamp to the current moment.
3. Node-to-Node Interlocking: When the worker moves to the next machine (PC-2 on Floor 1), the server calculates the time-of-flight relative to the scan of PC-1, not the lobby.
4. Floor Hopping Checks: When moving from PC-2 (Floor 1) to PC-3 (Floor 2), the database checks the specific physical relationship between those two assets. It knows that traveling between Floor 1 and Floor 2 requires a minimum elevator window of \(15\text{ seconds}\).
By chaining each asset scan to the one immediately preceding it, the system forms an unbreakable, dynamic track of physical presence that protects the entire building seamlessly from a single initial lobby initialization.



5. Component 4: Active Task Verification (The Step-by-Step Interlock)
A remaining loophole in time tracking is the "Ghost Work" scenario: a worker successfully traverses to Floor 3, scans the authentic QR code on a PC to pass the initial Time-of-Flight speed trap, but then places their phone flat on a desk to sleep or slacks off for 40 minutes while letting a company task timer run in the background to pad their billable hours.
To resolve this without tracking invasive background phone sensors, the system splits standard maintenance checklists into a Task Checklist Interlock.

[Scan Asset QR] ──► [Task Step 1] ──► [Task Step 2] ──► [Task Step 3] ──► [Final Submit]
                      (Min: 60s)         (Min: 120s)        (Min: 45s)      Task Approved

The Interactive Gate Architecture
Instead of offering a singular "Start" and "Complete" button interface, the software breaks down a 10-minute technical audit into small, mandatory interactive milestones inside the UI (e.g., Step 1: Check power cable connectivity, Step 2: Blow dust out of internal cooling fans, Step 3: Boot system and verify diagnostic logs).
* The Micro-ToF Check: The server logs individual timestamps when the user ticks off each separate step.
* Natural Pacing Enforcement: The system applies tiny minimum time rules to every single step based on real manual workflows. For instance, cleaning a dust filter physically requires at least 60 seconds of human effort.
* Catching the Macro-Cheat: If a worker attempts to bypass the checklist by instantly rapid-firing their fingers across Checkbox 1, Checkbox 2, and Checkbox 3 within 3 seconds, the server calculates the sub-second gaps between the steps, flags the pacing as humanly impossible for manual labor, and voids the submission.
